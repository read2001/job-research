# Job Research — 招聘公司调研归档

## 这个 skill 做什么

用户在求职，遇到一家公司想知道真实情况。skill 做三件事：

1. **（必要时）先建画像**：如果还没有候选人画像，先花几分钟把简历、求职偏好、经历访谈整理好，存进 `resume/meta/`。已有了就跳过。
2. **联网搜索**：用 `mcp__web-search-prime__web_search_prime` 做 3-4 轮搜索，挖出招聘简章不会写的真实信息（待遇水分、加班、裁员、员工吐槽）。
3. **匹配度分析 + 归档**：读画像，把搜索结果和候选人的背景、目标岗位交叉分析，在 `resume/jobs/<公司全称>/` 下生成三个文件。

## 路径约定

所有路径相对于**当前工作目录**，根目录是 `resume/`：

- **候选人元资料**：`resume/meta/`
  - `profile.md` — 经历画像（画像阶段写，research 阶段只读）
  - `preferences.md` — 求职偏好（同上）
  - `resume/` — 简历原件（PDF/DOCX/MD）
- **jobs 根目录**：`resume/jobs/`
- **公司目录**：`resume/jobs/<公司全称>/`（用全称，不用简称，避免歧义）

> 找不到 `resume/` 时先搜：`find . -type d -name resume`。都没有就在当前目录创建 `resume/meta/`、`resume/jobs/`。这是 skill 自己维护的目录，朋友从零开始也能直接用。

## 完整流程

### 前置第 1 步：画像检查（决定要不要先做画像）

research 真正开工前，先看候选人画像在不在。

1. 定位 `resume/` 目录（见路径约定）。
2. 检查 `resume/meta/profile.md` 是否存在、且不是空模板——要有候选人姓名、至少一段经历，而不是 `[姓名]` 这类没填的占位符。

- **有画像**（profile.md 真实存在）→ 直接跳到「第 0 步：快速预筛」。research 阶段 `resume/meta/` 只读，不改。
- **没有画像**（不存在 / 只是空模板）→ 先走下面的「画像流程」，把简历、偏好、经历访谈补齐写进 `resume/meta/`，再回到第 0 步。

> 强制更新画像：用户用 `/job-research profile`（或参数是 `profile` / `画像`）时，**即使已有 profile 也重跑画像流程**，且不进入 research。

### 画像流程（仅在缺失时执行；`profile` 参数时强制执行）

#### P1. 简历

请候选人提供简历。接受：

- 文件路径（PDF/DOCX/MD）→ 复制进 `resume/meta/resume/`
- 直接粘贴文字 → 存为 `resume/meta/resume/resume.md`

若 `resume/meta/resume/` 已有简历，确认是否直接用现有的，不必重收。收齐后简述：姓名、最新身份或届别、有几段经历。

#### P2. 求职偏好

用一句自然的话问：

> "你想找什么样的工作？说说目标岗位、城市、薪资期望，还有任何你想避开的情况。"

据回答存 `resume/meta/preferences.md`（模板见下）。漏了某项没关系，先存有的，以后能补。

#### P3. 经历访谈

按 `references/conduct-interview.md` 做一次对话式访谈，从最重要/最近的经历开始。每个经历问清：背景与性质、候选人做了什么、成果与指标（抓数字）、技术与方法、挑战与收尾。含糊就追问具体数字。

访谈后，按下面的模板写 `resume/meta/profile.md`。

画像完成后，简短汇报（建了哪些文件、覆盖几段经历、核心方向一句话），再继续 research。

### 第 0 步：快速预筛（省钱省时间）

读完招聘信息和候选人资料后，先做一次方向匹配预判：

- 这家公司的**主业**和候选人的**核心方向**是否沾边？
- 招聘信息里列出的**专业需求/岗位**是否包含候选人的专业？

如果**明显不沾边**，不要急着开搜。直接告诉用户："这家公司的主业是 XX，和你的方向匹配度看起来很低，要我继续深挖确认吗？" 让用户决定是否值得花搜索成本。

如果**沾边或不确定**，进入正常流程。

### 第 1 步：确认公司身份

从用户输入提取公司名。如果用户贴的是招聘海报/JD 文字，先识别全称。

关键区分（常见易混淆）：
- 集团总部 vs 子公司 vs 研究院（待遇差距大，必须分清）
- 官方全称 vs 简称（目录用全称）
- 同名/近名单位要分清是两家不同的单位

如果不确定，直接问用户："全称是 XX 对吗？"

**同时检查招聘信息是否完整。** 如果用户给的招聘信息缺关键内容（岗位、专业需求、待遇、投递方式），提示用户："这份招聘信息没有 XX，我去招聘系统/官网看看能不能补上？" 然后尝试用搜索补全。

### 第 2 步：读候选人资料

读 `resume/meta/profile.md` 和 `resume/meta/preferences.md`，掌握：
- 候选人的学校、专业、学历
- 技术方向和核心能力
- 求职偏好（目标岗位、城市、薪资期望、硬性要求）
- 项目/工作经历（用于判断和岗位的匹配度）

**research 阶段不要修改 meta 目录下的任何文件。** 若 preferences.md 缺失，提醒用户可随时用 `/job-research profile` 补上，但不必中断当前调研。

### 第 3 步：联网搜索（标准 3-4 轮）

用 `mcp__web-search-prime__web_search_prime`，这个比普通 web search 质量更好。参数默认：`content_size: "high"`, `location: "cn"`。

按这个顺序搜，每轮根据上一轮结果调整：

**第 1 轮：公司基本信息 + 待遇**
```
搜索词：<公司全称> 待遇 应届生 薪资 户口
```
要搞清楚：归属集团、成立时间、单位性质、招聘待遇区间。

**第 2 轮：员工真实评价**
```
搜索词：<公司全称> 怎么样 评价 加班 离职 真实情况
```
找主流评价平台（如知乎、脉脉、看准网、offershow）上的在职/离职员工说法。

**第 3 轮：负面信息 + 深层问题**
```
搜索词：<公司全称> 裁员 降薪 问题 缺点 坑
```
主动找负面。招聘广告都是好话，真实价值在负面上。

**第 4 轮（按需）：针对性深挖**
根据前面三轮发现的线索，针对性搜。比如：
- 发现是子公司 → 搜"XX集团 子公司 区别 编制"
- 发现是央企/国企 → 搜"央企 劳务派遣 正式工 区别"
- 发现待遇存疑 → 搜"XX 月薪 实发 到手 倒挂"

搜索原则：
- **时间优先近 1 年**（`search_recency_filter: "oneYear"`），公司情况变化快
- **深度搜索用 high**（`content_size: "high"`），要细节不要标题
- **搜到的关键页面用 `mcp__web-reader__webReader` 深读**，尤其是长回答
- **找不到信息本身就是信息**——搜不到说明公司小/新/没存在感，要如实告诉用户

### 第 4 步：读到的页面深挖

对搜索结果中信息量大的页面，用 `mcp__web-reader__webReader` 读取全文。重点读：
- 问答页（多个回答一起看，分辨内部人 vs 外行）
- 评价平台的具体评价
- 招聘公告原文（对比官方口径）

**不要只看搜索摘要就下结论。** 摘要经常断章取义。

### 第 5 步：综合分析

整合所有信息，形成判断。分析框架：

**A. 单位基本情况**（表格）：全称、上级、成立时间、性质、办公地、注册资本

**B. 招聘待遇（官方版）**（表格）：薪资、户口、福利

**C. 真实情况（招聘简章不写的）**：这是核心价值。包括：
- 用工形式（正式合同/劳务派遣/外包，要分清）
- 薪资结构（基本工资占比、年终占比、实际到手 vs 纸面总包）
- 稳定性真相（自负盈亏？有裁员？业务线在收缩？）
- 加班、管理、培养体系成熟度
- 负面评价（如实呈现，标注来源和时效）

**D. 真正的好处**：公平起见，优势也要列清楚

**E. 和候选人的匹配度**：结合 profile.md，分析：
- 专业/学历是否对口
- 候选人的技术方向和岗位是否匹配
- 薪资/城市/稳定性是否符合候选人期望
- 有哪些风险点（技术成长、技能折损、现金流、长期不确定性）

**F. 投递建议**：
- 建议投 / 谨慎投 / 不建议投
- 作为什么梯队的选择（首选/备选/保底）
- 面试时务必问清的问题清单

### 第 6 步：归档

创建 `resume/jobs/<公司全称>/`，生成三个文件：

**文件 1：`招聘信息原文.md`**
用户提供的招聘信息整理成结构化 markdown。如果用户给的是图片转文字，注明"可能有错字"。岗位用表格呈现。

**文件 2：`分析与决策.md`**
第五步的完整分析。用 markdown，表格 + 分段。开头写：
```
> 分析日期：YYYY-MM-DD
> 候选人：<姓名> · <学校> · <专业> · <届别>
> 目标方向：<岗位>
```
文末附"信息来源"（所有搜索到的关键 URL，markdown 链接格式）。

**文件 3：简历副本**
从 `resume/meta/resume/` 找到简历文件，复制到公司目录。文件名优先级：
1. 招聘信息里明确要求的投递命名格式（如"姓名+院校+专业+学历+岗位序号"）
2. 没有要求时，用 `<姓名>+<学校简称>+<专业>+<学历>.<原后缀>`

姓名/学校/专业一律从 profile.md 取，不要猜。注意：如果命名格式里有"岗位序号"这种投递时才填的字段，简历文件名不带序号（保持简历可复用），投递时再 rename 加上。

### 第 7 步：汇报

归档完成后，给用户一个简短汇报：
- 创建了哪个目录、几个文件
- 分析的核心结论（投不投、为什么）
- 有没有需要用户补充的信息（比如岗位序号还没定）

## 写作风格

- **平实直白，说人话**。不要公文腔，不要"赋能""抓手""闭环"这类词
- **负面信息敢写**。这是这个 skill 的核心价值。招聘广告都是好话，用户需要的是另一面。但也要公平——优势同样要列
- **标注信息来源和时效**。"2023年的评价"、"某平台高赞"，让用户自己判断可信度
- **给明确建议**。别只罗列信息不给结论。用户要的是"投不投"的判断
- **区分确定和不确定**。"确定的是…，不确定的是…"，不要把猜测说成事实

## 边界

- **research 阶段 meta 只读**：`resume/meta/` 下的文件只读不改，除非用户明确要求或正在跑画像流程
- **画像阶段才写 meta**：只有画像流程（缺失时或 `profile` 参数）会写 `resume/meta/`
- **不改已有公司目录**：如果 `resume/jobs/<公司>/` 已存在，先问用户是覆盖还是追加
- **不替用户做最终决定**：给建议，但最终投不投用户自己定
